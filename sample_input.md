# Ecom Ops Agent 面试演示说明

## 1. 这个 Demo 解决什么问题

电商运营每天都需要查看 SKU、销售、库存和订单数据，并快速回答几个问题：

- 今天整体销售表现怎么样？
- 哪些 SKU 热销？哪些 SKU 滞销？
- 有没有库存不足或库存积压？
- 有没有退款、延迟发货、待付款或售后异常？
- 哪些问题需要同步给供应链、客服、销售或商品团队？
- 今天的日报、飞书通知和复盘 SOP 应该怎么写？

Ecom Ops Agent 的目标不是做一个静态数据看板，而是模拟“AI Agent 如何参与电商运营流程”：自动读取数据、计算指标、识别异常、生成运营日报、沉淀 SOP，并保留人工确认节点。

## 2. 四张表分别代表什么

### SKU 主数据表

`sku_master.csv` 代表商品基础信息，包括 SKU 编码、商品名称、类目、规格、平台、价格、上架状态和商品负责人。

它用于回答：这个 SKU 是什么商品？属于哪个类目？由谁负责？在哪个平台销售？

### 销售数据表

`sales_daily.csv` 代表按天、平台、渠道和 SKU 记录的销售表现，包括访客、点击、订单、销量、GMV、退款和广告花费。

它用于计算 GMV、销量、订单数、转化率、退款率、广告 ROI 和 SKU 排名。

### 库存数据表

`inventory_daily.csv` 代表每个 SKU 的库存快照，包括可售库存、锁定库存、在途库存、安全库存、补货周期和库存状态。

它用于识别库存不足、热销缺货和滞销积压。

### 订单数据表

`orders.csv` 代表订单明细，包括订单状态、履约状态、退款状态、发货耗时和用户问题。

它用于识别延迟发货、退款异常、待付款异常、履约异常和售后异常。

## 3. 系统如何识别异常

系统使用规则引擎完成第一版异常识别：

### 库存风险

- 可售库存低于安全库存，或预计可售天数低于补货周期 → 高风险库存不足。
- 预计可售天数低于补货周期 + 3 天，或可售库存接近安全库存 → 中风险库存不足。
- 热销 SKU 的预计可售天数低于补货周期 + 3 天 → 热销缺货风险。
- 近 7 日日均销量低于 3 件，且预计可售天数超过 60 天 → 滞销积压。

### 订单异常

- 发货耗时大于 3 天 → 延迟发货。
- SKU 退款率高于 5% → 退款异常。
- SKU 待付款订单占比高于 15% → 待付款异常。
- 已付款但长期待发货 → 履约异常。
- 用户问题集中出现“质量问题”“物流延迟”“缺件”“不会使用” → 售后异常。

## 4. AI 在流程中负责什么

当前版本没有调用真实大模型 API，而是用规则 + 模板模拟 AI Agent 的输出逻辑。AI 在流程中扮演的角色是：

1. 把分散的数据表转换成经营指标。
2. 把指标异常转换成业务风险描述。
3. 把风险描述转换成不同角色能执行的动作建议。
4. 把分析结果组织成运营日报。
5. 把日报压缩成适合飞书群同步的通知文案。
6. 把重复出现的问题沉淀成 SOP 和复盘字段。

后续接入大模型后，可以让模型负责更自然的原因分析、跨表归因、复盘总结和多角色沟通文案生成。

## 5. 为什么需要人工确认

电商运营中的异常判断往往会影响投放、补货、客服话术和销售节奏，不能完全依赖自动化结论。

例如：

- 系统识别某个 SKU 存在断货风险，但供应链可能已经安排紧急调拨。
- 系统识别某个 SKU 退款率偏高，但退款可能来自一次物流事故，而不是商品质量问题。
- 系统建议降低广告投放，但该 SKU 可能正在参与平台活动，需要运营负责人确认。

因此，这个 Demo 保留“AI 总结 → 人工确认 → 飞书通知 → SOP 沉淀”的流程设计。AI 负责减少重复整理工作，人负责业务判断和最终决策。

## 6. 这个工具如何迁移到飞书、多维表格或企业内部智能体

可以按以下方式迁移：

### 迁移到飞书多维表格

- 将四张 CSV 表替换为飞书多维表格数据源。
- 使用飞书开放平台 API 定时拉取 SKU、销售、库存和订单数据。
- 将异常结果写回多维表格，形成“待处理异常池”。

### 迁移到飞书机器人通知

- 将 `report_generator.py` 生成的飞书通知文案推送到飞书群机器人。
- 根据风险等级控制通知频率，例如高风险立即通知，中风险进入日报。
- 在通知中加入负责人、SKU、风险类型和建议动作。

### 迁移到企业内部智能体

- 将规则计算模块保留为确定性工具。
- 将日报生成、原因分析和 SOP 总结替换为大模型调用。
- 增加 RAG 知识库，让智能体读取历史复盘、商品手册、客服话术和库存策略。
- 增加人工确认按钮，把 AI 建议转化成任务流。
- 增加权限和日志，记录谁确认、谁处理、何时关闭异常。

## 7. 面试演示建议

演示时可以按以下顺序：

1. 说明项目不是普通看板，而是电商运营 AI 工作流 Demo。
2. 展示左侧 CSV 上传入口，并说明无上传时使用内置示例数据。
3. 展示四张表预览，说明数据来源和字段含义。
4. 展示核心指标卡片，说明系统自动计算 GMV、销量、转化率、退款率和 ROI。
5. 展示 SKU 分析区，重点讲 GMV TOP、销量 TOP、退款率 TOP 和库存风险 TOP。
6. 展示库存风险区，说明热销缺货和滞销积压如何被识别。
7. 展示订单异常区，说明延迟发货、退款异常、待付款异常和售后异常如何被识别。
8. 点击“生成 AI 运营日报”，展示结构化日报。
9. 展示飞书通知文案，说明如何迁移到飞书机器人。
10. 展示 SOP / 复盘沉淀建议，强调 AI Agent 的价值不只是回答问题，而是沉淀业务流程。
